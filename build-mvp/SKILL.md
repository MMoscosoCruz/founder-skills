# build-mvp

**Purpose**  
Help founders build the smallest possible thing that generates real learning, while avoiding the common mistake of overbuilding too early.

**When to use this skill**  
- You have some validation but are tempted to add more features
- You want to ship quickly and learn from real usage
- You feel pressure to make the product look polished before showing it

**Core Principles**  
- If you’re not embarrassed by your first version, you launched too late.
- An MVP is a learning tool, not a finished product.
- Being small is an advantage early on.
- Most features you think are necessary are not.

**Decision Framework**  
Before adding any feature, ask:

1. Does this directly help test the core assumption?
2. Can I learn what I need without building this?
3. Will this slow me down significantly?

**Action Steps**  
1. Define the single most important user behavior you need to test.
2. Build only what is required for that behavior to work end-to-end.
3. Include the absolute minimum (auth, core flow, basic UI).
4. Ship within 2–3 weeks maximum.
5. Get it in front of the people you already spoke to during validation.

**What Good Looks Like**  
You feel slightly embarrassed when showing it. Real users engage with it and you collect concrete feedback. You’re learning faster than you’re building.

**Red Flags & Common Mistakes**  
- Adding nice-to-have features before the core loop works.
- Optimizing for looking professional instead of learning.
- Waiting until the product feels “ready” before showing it to anyone.

**Example Output**  
A well-scoped MVP includes only:
- Basic login
- One core user flow that solves the main pain
- Enough UI to be usable
- Payments (if charging from day one)

Everything else is delayed until real usage data justifies it.
